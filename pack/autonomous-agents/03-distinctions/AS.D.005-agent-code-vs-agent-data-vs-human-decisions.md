# AS.D.005: Код агента ≠ Данные агента ≠ Решения человека

## Различение

| | Код агента (instrument) | Данные агента (governance) | Решения человека (governance) |
|-|------------------------|---------------------------|-------------------------------|
| **Что** | Промпты, скрипты, agent-card, trajectory cache | Черновики, отчёты, находки, extraction-reports | Утверждённые планы, подтверждённые captures |
| **Где** | DS-autonomous-agents | DS-agent-workspace | DS-my-strategy |
| **Частота изменений** | Редко (версии кода) | Ежедневно (ночные циклы) | При утверждении |
| **Автор** | Разработчик | Агент | Человек |
| **Тип репо** | DS/instrument | DS/governance | DS/governance |

## Почему это важно

Без разделения:
- Код агентов раздувается ежедневными данными (git history загрязняется)
- Непонятно кто автор артефакта (агент или человек?)
- При масштабировании (N агентов) данные всех агентов смешиваются с кодом

## Критерий проверки

**Тест:** Кто автор артефакта?
- Агент → DS-agent-workspace
- Человек → DS-my-strategy
- Разработчик (код/конфиг) → DS-autonomous-agents

## Граничные случаи

### Trajectory cache — код или данные?

**Код.** Trajectory cache (rules.md, accepted.md, rejected.md) влияет на поведение агента — это его «обученные веса». Без trajectory агент деградирует. Живёт в instrument-репо.

### Черновик Стратега — данные агента или решение человека?

**Данные агента** (до утверждения). Черновик DayPlan от автономного Стратега → DS-agent-workspace/strategist/. После проработки человеком → утверждённая версия → DS-my-strategy. Два разных артефакта: raw (что предложил агент) vs approved (что решил человек).

## Pipeline

```
Агент создаёт артефакт
  → DS-agent-workspace/{agent}/YYYY-MM-DD/
  → Человек ревьюит
  → Утверждённая версия → DS-my-strategy (или Pack)
  → Черновик остаётся в workspace (аудит, обучение)
```

## Конвенция размещения

Каждый агент пишет в `DS-agent-workspace/{agent-name}/YYYY-MM-DD/`. Имя папки = имя роли (lowercase, kebab-case).
